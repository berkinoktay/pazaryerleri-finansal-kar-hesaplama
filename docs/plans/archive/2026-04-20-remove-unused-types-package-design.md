# Remove Unused `@pazarsync/types` Package — Design

**Date:** 2026-04-20
**Status:** Proposed
**Scope:** Small (single refactor commit)

## Context

`packages/types/` was introduced early in the project to be the single
source of truth for types shared between `apps/api` and `apps/web` —
API request/response contracts, domain enums (`Platform`, `OrderStatus`,
`MemberRole`, …), and marketplace integration types.

That role was taken over by two later migrations, neither of which
cleaned up `packages/types/` as part of the same change:

1. **Prisma 7 generator** (`packages/db`) emits TypeScript types for
   every enum declared in `schema.prisma`. Backend imports them via
   `import { OrderStatus } from '@pazarsync/db'`. This makes
   `packages/types/src/enums.ts` a duplicate, with drift risk if one
   side is updated and the other is not.

2. **OpenAPI pipeline** (`packages/api-client`, PR landed 2026-04-16
   — see `docs/plans/2026-04-16-api-docs-design.md`) generates typed
   request/response shapes from backend Zod validators. Frontend
   imports them via `components["schemas"]["..."]`. This makes
   `packages/types/src/api.ts` (`ApiResponse`, `PaginatedResponse`,
   `ApiError`) redundant against a stricter, drift-proof source.

Both migrations replaced the hand-curated package with code-generated
types that derive from a single source (Prisma schema and Zod schemas
respectively). `packages/types/` has been orphaned since.

## Evidence

Grep across the whole repository confirms **zero import sites** for
`@pazarsync/types` in application source code:

```
$ grep -r "@pazarsync/types" --include="*.ts" --include="*.tsx"
# Only hits:
#   packages/types/package.json           (self)
#   packages/types/README.md              (self)
#   apps/web/package.json                 (dependency declaration)
#   apps/api/package.json                 (dependency declaration)
#   apps/web/next.config.ts               (transpilePackages entry)
#   pnpm-lock.yaml                        (auto-generated)
#   docs/ARCHITECTURE.md                  (stale prose reference)
#   docs/plans/2026-04-16-api-docs-design.md (historical planning doc)
#   apps/web/CLAUDE.md                    (stale prose reference)
#   CLAUDE.md                             (stale prose reference)
#   packages/types/README.md              (self)
```

No `.ts`/`.tsx` file in `apps/` or `packages/` actually imports a symbol
from `@pazarsync/types`. The package exists only as a transitive
dependency entry in `package.json` files and stale documentation.

## Decision

Remove the package entirely. Type sharing across the frontend/backend
boundary is already accomplished by two mechanisms that do not depend
on `packages/types/`:

- **Enums and domain row shapes** → `packages/db` (Prisma-generated)
- **API request/response contracts** → `packages/api-client` (OpenAPI-generated)

Marketplace adapter types (`MarketplaceOrder`, `MarketplaceProduct`,
`MarketplaceSettlement`) are not re-homed in this change because no
adapter code exists yet. When the background-sync milestone begins,
those types will be created from scratch in
`apps/api/src/integrations/marketplace/types.ts`, as already specified
by `apps/api/CLAUDE.md` § Marketplace Adapters (Strategy Pattern) and
`docs/ARCHITECTURE.md` § 7.

## Non-Goals

- **Not** creating `apps/api/src/integrations/marketplace/types.ts`
  ahead of the adapter implementation. YAGNI — when the adapter PR
  arrives, git history of this commit preserves the original drafts.
- **Not** touching `supabase/functions/` scaffolds or empty SQL files
  in `supabase/sql/`. Those are covered by a separate scope decision
  (see brainstorming notes; flagged as "Scope B/C" there).
- **Not** editing historical planning documents under `docs/plans/*`.
  They record past state and should remain accurate to their date.
- **Not** renaming or restructuring `packages/db`, `packages/utils`, or
  `packages/api-client`. Those are actively used and correctly placed.

## Verification Plan

Run in order from repo root:

```bash
pnpm install          # lockfile regen, no errors
pnpm typecheck        # no broken imports anywhere
pnpm lint             # clean
pnpm build --filter web   # critical: transpilePackages changed
pnpm build --filter api   # sanity check
pnpm check:all        # full pre-commit gate (typecheck + lint + unit + format)
```

The build step for `apps/web` is the most load-bearing check: an
incorrect `transpilePackages` array can cause Next.js to fail at module
resolution during bundling. If `pnpm build --filter web` succeeds, the
package has been cleanly severed.

No integration tests need to run for this change (no DB or route
behavior touched), but `pnpm check:full` can be run before PR open for
extra confidence.

## Rollback

Single commit, single revert. Git history preserves the contents of
`packages/types/src/*` (notably `marketplace.ts`) for reference when
the background-sync milestone needs marketplace adapter type drafts.

## Risk Assessment

- **Drift risk from removal**: none — zero import sites today.
- **Future cost**: when marketplace adapters arrive, types will be
  written fresh in `apps/api/src/integrations/marketplace/types.ts`.
  Drafts are preserved in git for reference.
- **CI risk**: `pnpm install` lockfile regen must be committed
  atomically with the `package.json` edits. If they diverge, CI
  `pnpm install --frozen-lockfile` will fail. The verification plan
  catches this locally before commit.
